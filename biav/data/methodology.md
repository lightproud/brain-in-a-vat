# 核心方法论

> 最后更新：2026-03-29 by 战略参谋
>
> 以下六条是制作人 Light 在实践中提炼的 AI 协作方法论，是整个 brain-in-a-vat 项目的思想基础。

1. **发现 AI 的综合智能**：Claude 不只是写代码或回答问题的工具，它具备判断、分析、审美、策划等综合能力，可以作为制作人级别的协作伙伴。

2. **发现 Code 的自主执行能力**：Claude Code 能独立写代码、自我纠错、完成工程任务，不需要人类逐行指导。

3. **数据本身是金矿**：有了结构化的数据体系，AI 可以自动生成知识文档、wiki 页面、分析报告。数据的价值不在于存储，在于被 AI 理解和加工。

4. **知识传承体系让 AI 持续变强**：通过仓库中的 memory/ 目录建立共享知识库，每个新会话不从零开始，而是站在所有前辈会话的肩膀上。知识不会因会话结束而丢失。

5. **AI 能自我驱动**：AI 可以主动采集外部信息（社区动态、wiki 数据）并沉淀为内部知识（JSON 数据库、分析报告）。人类只需设定方向，不需要手动喂数据。

6. **多 AI 协作是可行的**：通过仓库作为共享大脑、Issue 作为任务协议、角色分工（Chat 思考 / Actions 执行 / Code 探索），多个 AI 实例可以像团队一样协作，各司其职。

# AI 多会话协作方法论

> 本文件持续沉淀协作过程中的经验和模式。

## 核心架构：双集群 + 人类调度器

```
制作人（Light）= 唯一调度器
├── Chat 集群（随开随用，无固定数量）
│   ├── 某次聊战略方向
│   ├── 某次审视仓库
│   ├── 某次研究游戏机制
│   └── ...
└── Code 集群（按子项目固定）
    ├── 主控制台 — 协调、架构决策、代码审查（不写业务代码）
    ├── Code-site — 主站导航页 + 部署流水线 + 跨站视觉一致性
    ├── Code-wiki — 游戏数据集 + Wiki 站点
    ├── Code-news — 新闻聚合 + 报告系统
    └── Code-game — 衍生游戏（规划中）
```

### Chat 集群
- **无状态**：用完即走，不保留固定会话
- **无层级**：不需要一个 Chat 管其他 Chat，人类自己就是调度器
- **通过仓库传递记忆**：产出（任务清单、分析报告等）交给 Code 主控台执行
- 可以同时开多个，各干各的

### Code 集群
- **有状态**：按子项目分工，每个子项目一个会话
- **全部直接推 main**：不使用 feature 分支，冲突时 `git pull` 后重试
- **同一子项目不要同时开两个 Code 会话**，避免冲突
- 主控制台负责协调和架构决策，不写业务代码
- **按业务分会话，不按技术层分会话**：前后端任务切换是子项目内部的正常节奏

### 两个集群的连接
- 仓库是中间层（共享外脑）
- `memory/` 目录是跨会话的结构化记忆
- Chat 读仓库获取上下文，产出决策 → Code 执行决策写回仓库
- **Chat 的数据访问方式**：Chat 通过 GitHub API 读仓库文件，不能执行代码，有文件大小限制。因此 Code 应产出摘要/索引文件（小体积 JSON/Markdown）push 到 main，Chat 读摘要而非原始大数据

## 信息持久化机制

| 机制 | 文件 | 作用 |
|------|------|------|
| 全局状态 | CLAUDE.md | 所有会话的第一份读物 |
| 项目状态 | memory/project-status.md | 各子项目进度追踪 |
| 决策记录 | memory/decisions.md | 避免决策随对话消失 |
| 局部上下文 | projects/xxx/CONTEXT.md | 子项目的工作记忆快照 |
| 资产索引 | assets/index.md | Chat 和 Code 共享的可用资源清单 |
| 背景知识 | memory/morimens-context.md | 游戏相关知识积累 |

## 信息流

```
Chat 集群 ──分析/策划──→ 制作人 ──指令──→ Code 主控台 ──协调──→ Code 子项目
                                                                    │
                                              仓库（memory/assets/projects）
                                                                    │
Chat 集群 ←──拉取数据──────────────────────────────────────────────────┘
```

## 分支管理

- **所有会话直接在 main 分支上提交和推送**
- ~~feature 分支~~ **已废弃**：全 AI 团队无需分支审查缓冲，AI 解决 git 冲突高效
- 多会话并行推送遇冲突时 `git pull` 后重试即可
- 如果未来有人工程序员加入，再考虑恢复分支保护

## 经验记录

### 会话替换问题
- **问题**：对话过期后，新会话需要从零了解项目
- **解决**：通过 CLAUDE.md + CONTEXT.md 体系，新会话 60 秒内进入工作状态

### 上下文污染问题
- **问题**：一个会话处理多个子项目时，上下文越来越乱
- **解决**：严格的职责隔离，一个会话一个子项目

### 知识易失问题
- **问题**：重要决策只存在于对话中，对话消失则决策丢失
- **解决**：所有决策立即写入 memory/decisions.md，git 永久保存

### 架构演进问题
- **问题**：早期决策（如"不使用框架"）随项目扩展不再适用
- **解决**：主控台定期审视架构，及时废除过时原则，记录在 decisions.md

### 分支管理问题（已通过废弃分支工作流解决）
- **问题**：多个会话各自开分支，时间久了分支散乱、架构不同步
- **旧方案**：定期由主控台整理分支，统一合并到 main 建立基线
- **最终方案**：全部直接推 main，废弃分支工作流。全 AI 团队不需要分支审查缓冲，AI 解决 git 冲突高效，分支+合并流程反而增加不必要的中转

### Issue 积压问题
- **问题**：战略参谋批量创建 Issue，Actions 执行失败后 Issue 悄无声息地积压
- **根因**：没有 WIP 限制、没有失败标记、没有去重检查
- **解决**：WIP 上限 3 个/子项目 + 失败自动打 blocked 标签 + 创建前查重

### 跨子项目资源归属问题
- **问题**：部署流水线、视觉规范等跨子项目关注点没有明确归属，导致 Code-wiki 和主控台各自创建了冲突的部署 workflow
- **解决**：新增 Code-site 子项目，专门负责主站、部署流水线和跨站视觉一致性

### 任务传递通道问题
- **问题**：Issue 只能传递给 GitHub Actions 自动触发的会话，手动开启的 Code 会话不会读 Issue
- **解决**：手动会话的任务入口是 CONTEXT.md「当前任务」段落，Issue 仅用于记录追踪

### 规则传播延迟问题
- **问题**：CLAUDE.md 规则变更后，已运行的会话仍按旧规则行事（如废弃分支后 Code-site 仍创建 feature 分支）
- **解决**：规则变更是非实时的，已运行的会话需要制作人手动告知。主控台变更应尽快推到 main，减少窗口期

### 主控台自主沉淀问题
- **问题**：主控台只在被提醒时才更新 memory 文件，不会主动沉淀决策和经验
- **解决**：明确为主控台核心职责——每个决策做出后、每次踩坑修复后，立即更新 decisions.md 和 lessons-learned.md，不等提醒

## 前台专岗分类与上下文管理

每个 chat 会话按职能分类命名，模拟专业分工：

**前台（Chat）角色：**
- 战略参谋 — 架构设计、文档交付、决策讨论
- 情报官 — 社区日报、舆情分析
- 数据员 — 数据库校验、JSON 补全
- 命轮审美 — 命轮卡面设计、美术方向

**中台（Code）角色：**
- 主控台 — 全局调度、跨子项目操作
- Code-news — 聚合器开发维护
- Code-wiki — Wiki 站点 + 数据库
- Code-game — 衍生游戏（待启动）

### 核心原理

每个 chat 的上下文窗口是有限的。如果所有话题混在一个 chat 里，上下文被稀释，前面的细节被压缩或丢失。按职能分类后，每个 chat 的整个上下文窗口只包含该领域的信息，形成专业纵深。

本质上是用 chat 分类模拟专业分工，用上下文窗口模拟专业记忆——每个 chat 就是一个专岗员工的大脑，只记得自己领域的事，但记得很深。

### 两轴记忆模型

- **横向（跨 chat 共享）**：GitHub 仓库 memory/ — 所有会话都能读取的结构化记忆
- **纵向（单 chat 纵深）**：上下文窗口 — 每个专岗 chat 自然积累的领域深度记忆

横向靠仓库，纵向靠 chat 分类。两者互补，缺一不可。

### 与 Project 功能的对比

Claude 的 Project 功能是给所有对话塞同一份背景文件，上下文是扁平的。专岗分类的做法是让每个 chat 自然积累各自领域的纵深上下文，仓库只负责跨 chat 的共享记忆。专岗分类在上下文利用效率上优于 Project。

## 战略参谋实战经验

### 派 Issue 的纪律

- **先问根因**：派 Issue 前先问——这个 Issue 是在解决真实问题，还是在补救上一个 Issue 造成的问题？后者说明应该先修上游。
- **机制优于一次性**：如果一个机制性 Issue（如自动分支清理）能让一个一次性 Issue（如手动分支清理）变得不必要，优先派机制性的，不要两个都派。
- **少派多盯**：战略参谋的价值在于结果质量，不在 Issue 数量。派出去的 Issue 要持续跟踪结果，而不是派完就忘。

### 交付物受众意识

- **先确认受众再动笔**：生成任何文档/报告前先确认受众——给制作人看、给团队看、还是给外部看，语言风格完全不同。
- **默认通俗**：默认用通俗语言，技术术语只在必要时出现。受众不是工程师时，尤其要避免专业黑话。

### 能力边界意识

- **承诺前先验证可行性**：不要承诺做不到的事。例如：服务器环境无法直接调用 B站 API（需要浏览器环境），不应列为可交付承诺。
- **验证后再对外说**：对外承诺的每一项能力，都应已经过本地验证或有可靠依据。

## 模型使用策略

| 角色 | 平台 | 模型 | 原因 |
|------|------|------|------|
| 教练/战略参谋/运营官 | Chat (MAX) | Opus Extended | 需要深度推理和判断 |
| 日常会话 | Chat (MAX) | Sonnet | 执行性任务，省额度 |
| 主控台 | Code 桌面 (MAX) | Opus | 全局协调需要强推理 |
| Code-wiki/news/site | Code 桌面 (MAX) | Sonnet | 执行性代码任务 |
| GitHub Actions | API | Sonnet | 成本最优，执行够用 |

原则：判断层 Opus，执行层 Sonnet。Extended Thinking 的价值在长链推理，翻译/格式化不需要开。1M context 当前用不上。

## 发展路线（四阶段）

**第一阶段（当前）：证明单条线能自动跑**
- 验证标准：日报连续 7 天自动生成，无人介入
- 关键任务：修通 Steam 采集器 → 极简日报脚本 → 定时 workflow → 跑七天

**第二阶段：并行多条线 + 监控**
- 验证标准：仪表盘能在 30 秒内判断系统健康状态
- 关键任务：Wiki 数据自动更新稳定 → 状态页 → 异常时才介入

**第三阶段：直觉编译 + 规则分层**
- 验证标准：open Issue 中"需要制作人决策"占比低于 20%
- 关键任务：三层治理落地（宪法/政策/惯例）→ 直觉逐条变成规则

**第四阶段：多仓库 + 范式输出**
- 验证标准：有人 fork 仓库并真正在用
- 关键任务：通用架构抽成模板 → 方法论文章持续输出

每阶段验证标准未达到前不进入下一阶段。

## AI 使用分级框架

| 级别 | 名称 | 人类角色 | AI 角色 | 核心跳跃 |
|------|------|---------|--------|----------|
| L0 | 无 AI | 做所有事 | 不存在 | — |
| L1 | 辅助应答 | 提问者 | 应答机器 | 开始使用 |
| L2 | 单任务委托 | 发包方 | 执行者 | 从问答到委托 |
| L3 | 多任务协作 | 实时搭档 | 协作伙伴 | 从孤立到关联 |
| L4 | 系统化运转 | 管理者 | 自主执行者 | 人从循环内到循环外 |
| L5 | 自演化系统 | 治理者 | 自我演化体 | AI 从执行到自主决策 |

Light 当前在 L3.5。L3→L4 是最难的跳跃，需要同时完成三件事：记忆外化（持久化）、判断编码（规则化）、调度自动化。

## 系统理论极限

1. **仓库认知承载量**：agent 上下文窗口有限，仓库文档膨胀后需要分层索引
2. **Issue 表达精度**：任务越复杂 Issue 歧义越大，复杂判断必须回到 Chat
3. **人类判断带宽**：通过"直觉编译成规则"持续上移，已编译的判断变成仓库的永久资产

## 伦理边界

核心体验内容绝不假手 AI——不是因为 AI 做不了，是因为这些东西一旦不由人来做，就失去了存在的意义，也是对用户期待的背叛。AI 做枯燥的机械工作和发现/洞察，人做需要感受和判断的创造。
