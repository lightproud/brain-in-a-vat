# Discord 数据归档系统设计方案

> 最后更新：2026-03-30 by Code-news
> 供主控台审阅。Code-news 会话整理，待决策后实现。

## 背景

忘却前夜 Discord 服务器数据量远超预期——单个频道历史消息达 **76 万条**。纯 git 存储不可持续，需要设计分级存储 + 增量抓取方案。

---

## 存储架构

```
projects/news/data/discord/
├── channels/
│   ├── 综合讨论_a7982947/        ← 频道名_ID后8位（sanitized）
│   │   ├── 2026-03-29.jsonl      ← 按自然日分片，保留60天
│   │   └── 2026-03-28.jsonl
│   └── 官方公告_b1234567/
│       └── ...
├── activity_daily/
│   └── YYYY-MM-DD.json           ← 每日纯统计摘要，永久保留
├── monthly_reports/
│   └── YYYY-MM.md                ← Claude 生成的月报，永久保留
├── channel_index.json            ← ID → 名称/类型映射
└── state.json                    ← 增量抓取游标

GitHub Releases
└── discord-archive-YYYY-MM.tar.gz  ← 超出60天的原文全量
```

**分级规则：**
- Git 保留最近 60 天完整消息（当月 + 上月作缓冲）
- 每月 1 日触发归档：上个自然月数据 → 打包推 Releases + Claude 生成月报 + 删除 git 中该月 JSONL
- 每日统计摘要和月报永久留 git（Chat 可读）

**论坛频道：** 按帖子创建日存储，60天外归档。已知限制：帖子归档后的新回复丢失（极少见，可接受）。

---

## 抓取策略：双轨并行

每次 workflow 运行执行两件事：

```
Run 1：今天增量（新消息） + 历史偿还 2026-02
Run 2：今天增量           + 历史偿还 2026-01
Run 3：今天增量           + 历史偿还 2025-12
...
Run N：今天增量（历史追完，纯增量模式）
```

**保存频率：** 每处理完一个频道的一个自然月，保存一次 state，确保重启后从断点继续。

**运行时限：** 5 小时（GitHub Actions 6 小时上限内）。

## 已识别问题

### 严重（影响数据完整性）

| # | 问题 | 影响 |
|---|------|------|
| 1 | state 只记录"当前处理哪个月"，没记录月内进度（`last_historical_message_id`） | 月内被杀后整月重来 |
| 2 | 历史消息写入是 append 模式，月内重启后消息重复追加 | JSONL 出现重复行 |
| 7 | Workflow 无并发保护，6 小时触发可能与上次重叠 | 两个 job 同时写文件冲突 |

### 重要（功能缺失）

| # | 问题 | 影响 |
|---|------|------|
| 3 | ANTHROPIC_API_KEY 余额为零，月报生成无 fallback | workflow 失败 |
| 4 | 频道名含 emoji（如 🔸綜合討論）做目录名有 git 兼容风险 | 可能导致 git 异常 |
| 5 | 论坛频道历史按月过滤逻辑比普通频道复杂 | 实现难度高 |
| 6 | `_pending_threads` 只在内存，不持久化 | 重启后线程数据丢失 |
| 8 | `/guilds/{id}/members` 需开启 Server Members Intent | 成员数据 403 |

### 次要（已知限制）

| # | 问题 | 处理 |
|---|------|------|
| 9 | 历史起点未定义（从哪个月开始往回追） | 从 guild 创建时间推算 |
| 10 | 频道改名后目录分裂 | 接受，历史数据不断裂 |
