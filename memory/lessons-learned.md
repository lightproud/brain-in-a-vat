# 经验教训

> 本文件记录项目实践中踩过的坑和提炼的经验，供后续会话参考。
> 新会话请在遇到类似场景时优先查阅此文件。

## 工作流与协作

### L-001 分支长期不合并导致冲突
**教训**：子项目分支长期不合并到 main，随时间积累的差异会使合并变得痛苦。
**实践**：每完成一个可交付的功能单元就开 PR，不要等"做完了再合"。
**参考**：decisions.md 分支管理策略

### L-002 Issue 未及时关闭导致积压
**教训**：任务完成后未关闭 Issue，积压到一定程度需要专门清理（如本 Issue #47）。
**实践**：每个 Issue 完成后立即 comment 结果并 close，不留积压。

### L-003 会话上下文污染
**教训**：一个会话处理多个子项目，上下文越来越乱，产出质量下降。
**实践**：严格职责隔离，一个 Code 会话只负责一个子项目。

## 数据与代码

### L-004 早期决策随项目扩展失效
**教训**：早期"前端不使用框架"原则在 wiki 站点需求出现后已不适用，但惯性使用导致浪费时间。
**实践**：重要技术决策要定期回顾，过时原则立即废除并记录。
**参考**：decisions.md 2026-03-28 废除记录

### L-005 聚合器无空数据保护
**教训**：聚合器在所有数据源均失败时，会用空列表覆盖已有的 news.json，导致前端展示空白。
**实践**：写入前检查新数据是否为空；若为空则保留旧数据并记录警告。
**参考**：Issue #37，aggregator.py 修复

### L-006 数据 ID 不一致导致引用混乱
**教训**：早期角色 ID 使用拼音（rui_xi），后改为英文 slug（ruixi），中间过渡期引用混乱。
**实践**：一次性全量替换，不留 legacy 引用。
**参考**：decisions.md 角色 ID 英文 slug

## AI 协作

### L-007 新会话无法快速进入状态
**教训**：早期没有 CONTEXT.md 体系，新会话需要大量对话才能了解子项目状态。
**实践**：每个子项目维护 CONTEXT.md，记录"做了什么/待做什么/关键文件"。
**参考**：projects/*/CONTEXT.md

### L-008 决策只存在于对话中
**教训**：多次出现"之前说好的方案"因对话消失而无法追溯的情况。
**实践**：所有重要决策立即写入 memory/decisions.md，用 git 永久保存。

### L-009 过于激进的自动化未经验证
**教训**：Claude Code GitHub Actions 初次配置时，未经充分测试就接入 Issue 触发，导致早期 Issue 被错误执行。
**实践**：新的自动化管道先在隔离环境测试，确认安全策略（如 author 过滤）有效后再上线。
**参考**：decisions.md Issue 安全策略

### L-010 并行任务缺乏协调机制
**教训**：多个 Code 会话同时对同一文件（如 decisions.md）做修改，产生合并冲突。
**实践**：同一文件由唯一会话负责维护；跨会话共享文件修改由主控台统一协调。
