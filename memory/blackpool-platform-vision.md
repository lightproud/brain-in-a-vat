## 三、四种客户端定位（构想）

| 客户端 | 定位 | 场景 | 复杂度 |
|--------|------|------|--------|
| **钉钉机器人** | 碎片化写入 | 路上/开会/随手记决策 | 低（webhook） |
| **黑池 Web** | 深度工作台 | 看全貌、AI 多轮对话、排期调整、敏感数据 | 高 |
| **CLI** | 开发者入口 | biav_engine.py 直接跑，和 Web 共享后端 | 已实现 |
| **银芯** | 公开展示 | Wiki、日报、社区数据（只读消费） | 已有 |

Web 和 CLI 可能合并为一个应用（Web 内嵌终端），但这不是必须的。

## 四、结构化信息流替代 IM 群聊（构想）

### 问题

扁平团队 + 沟通透明 = IM 群里 5 个话题 10 分钟内交织。说完就沉，搜索靠记忆。

### 思路

IM 负责说话，黑池负责记住。钉钉机器人是桥梁：

```
钉钉群对话
  ├── @AI 某某方案定了     ← 触发：机器人提取、结构化、写入黑池
  ├── 正常闲聊             ← 不触发
  └── #标签 关键信息       ← 触发：自动归类
```

### 信息流条目数据模型

```json
{
  "id": "entry-001",
  "type": "decision | update | blocker | auto | task | milestone",
  "tag": "数值",
  "author": "策划A",
  "title": "洛水数值调整方案 A 确认",
  "body": "详细内容...",
  "status": "open | resolved | blocked",
  "refs": ["memory/decisions.md#洛水调整"],
  "source": "dingtalk:msg:xxx",
  "ts": "2026-04-05T09:00:00Z"
}
```

## 五、工单/排期整合（构想）

### 核心理念

工单不是填的，是聊出来的。对话中产生的任务，机器人提取后制作人确认即可。

### 工单 = 信息流的 task 类型

```json
{
  "id": "task-032",
  "type": "task",
  "tag": "数值",
  "title": "洛水数值调整（方案 A）",
  "assignee": "策划A",
  "status": "in_progress",
  "due": "2026-04-09",
  "subtasks": [
    {"title": "调整基础数值", "done": true},
    {"title": "内部测试", "done": false}
  ]
}
```

### 排期 = 工单的时间线视图

不需要独立排期工具。工单有 due + assignee + status，按时间排列就是排期。

## 八、如果要做，建议顺序

> 以下仅为讨论中的初步判断，不代表确定的执行计划。

1. **钉钉机器人 + 信息流写入** — 投入最小，团队马上能用
2. **信息流仪表盘（只读）** — 有了数据再做展示
3. **AI 对话集成** — biav_engine.py 接入 Web
4. **工单/排期** — 等信息流跑通再加
5. **Web + CLI 合并** — 最后优化体验
