## 零、项目本质重新定义

### 双系统架构

缸中之脑不是一个系统，而是两个：

| 系统 | 代号 | 定位 | 信息边界 |
|------|------|------|----------|
| **银芯** | 本仓库（brain-in-a-vat） | 公开信息层 | 仅使用公开可查阅信息：采访、Steam 页面、社区讨论、公开 PV、官方公告等 |
| **黑池** | 公司内部系统 | 内部信息层 | 内部设计文档、未发布内容、商业数据、内部决策记录等 |

**银芯的职责边界**：
- 公开信息的采集、结构化、展示
- AI 多会话协作方法论的验证与开源
- 面向社区的 Wiki、新闻聚合、衍生创作
- **绝对不触碰未公开的内部信息**

**银芯与黑池的关系**：
- 银芯是方法论的试验场——验证可行后，黑池复用同样的模式处理内部数据
- 银芯的所有成果开源，黑池的内容不公开
- 两者共享同一套 AI 协作架构（会话角色、memory 结构、知识捕获流程），但数据完全隔离
- 银芯像"公开的前厅"，黑池像"上锁的档案室"

### 核心定义

缸中之脑（银芯 + 黑池）是 B.I.A.V. Studio 的组织记忆系统。

银芯解决的核心问题是：如何用公开信息构建一个有价值的知识体系，同时验证 AI 协作方法论，使其可复用、可开源、可被其他团队采纳。

这意味着银芯的所有子项目优先级排序，都应该用同一个标准来判断：**它是否让公开信息的记忆更完整、更可用，同时为方法论提供可量化的验证？**

## 一、三条战略主线

### 主线 A：事实圣经（Fact Bible）

**目标**：把六年积累的游戏知识从人脑搬到仓库，结构化存储，任何 AI 会话都能即时调用。

**为什么是最高优先级**：
- 采访披露的信息量巨大（叙事三部结构、角色构思时间线、设计哲学、被砍机制、未公开角色），这些以前只存在于 Light 和霁月的脑子里
- 团队历史证明人员会流动——只有 QA 和作曲家从头到尾没走过
- 事实圣经同时服务 Wiki（对外展示）、日报（上下文理解）、新会话启动（快速对齐）、衍生游戏（设定基础）

**交付标准**：任何新启动的 AI 会话，读完 `memory/morimens-context.md` + `assets/data/` 后，能对 Morimens 的世界观、设计决策、团队历史、叙事结构做出准确陈述，无需额外搜索。

### 主线 B：自动化情报循环

**目标**：社区数据自动采集 → 日报自动生成 → 制作人 5 分钟掌握脉搏 → 省下的时间回到游戏开发。

**为什么是第二优先级**：
- 制作人确认高透明度社区互动是有意识的战略选择（弥补调研不足 + 运营人手 + 争取理解）
- 但这消耗大量时间。日报系统的终极价值 = 制作人不需要亲自刷社区也能掌握全貌
- 采访确认即将有宇宙恐怖经典作品联动——日报需要在联动前完成验证，否则错过第一次真实热度事件

**交付标准**：制作人连续 14 天主动看日报，觉得有用，不需要额外刷社区来补信息差。

### 主线 C：权威知识站点

**目标**：Wiki 成为 Morimens 最权威的公开信息源。制作人出品，数据最全，三语言覆盖。

**为什么是第三优先级**：
- 制作人身份赋予 Wiki 独特权威性（vs 社区自建 Wiki）
- 但权威性取决于内容质量。当前 189 页自动生成，数据完整度和准确度未经验证
- Wiki 是面向外部的"门面"，也是方法论开源的展示窗口

**交付标准**：搜索 "Morimens [角色名]" 时，本 Wiki 能出现在搜索结果前 3 页。角色页数据经制作人或社区抽查确认准确。

## 二、分阶段执行计划

| 任务 | 负责 | 状态 | 阻塞项 |

| 暂停无效 workflow | 战略中心 | ✅ 已完成 | 无 |

**Phase 0 出口标准**：日报覆盖 3 个有效数据源（Bilibili + Steam + Discord），仓库体积不再膨胀。

### Phase 1：记忆宫殿（4 月中旬 → 5 月底）

> 把知识从人脑搬到仓库。事实圣经 v1.0。

**1a. 数据录入（主线 A）**

| 任务 | 说明 |
|------|------|
| 采访数据二次结构化 | 从 `interview-2026-04.json` 提取角色信息→补充 `characters.json`；提取叙事结构→创建 `narrative-structure.json`；提取设计决策→创建 `design-decisions.json` |
| 制作人公开信归档 | 制作人信内容结构化存入 `assets/data/`，标注时间线、关键事件、承诺 |
| Wiki 数据质量基线审计 | 21 个 JSON 文件逐一抽查准确性，建立数据可信度评级 |
| 被砍机制 / 原始设计归档 | "代号：密境"的早期构想、疯狂量表等被砍机制的设计意图和砍掉原因，写入设计决策库 |

**1b. 情报循环验收（主线 B）**

| 任务 | 说明 |
|------|------|
| Stage 1 正式开始计时 | Phase 0 完成后，日报覆盖 3 源，开始 14 天验收 |
| 日报内容增强 | 利用事实圣经给日报加上下文（例如 Steam 评论提到某角色时，自动关联该角色的设计意图）|
| THPDom 效应监测原型 | 建立"高影响力社区内容"的自动标记机制——点赞/观看超过阈值的内容提升到日报头部 |

**1c. 知识捕获流程建立**

| 任务 | 说明 |
|------|------|
| 定义"知识进入仓库"的标准流程 | 制作人/主文案说了重要信息 → 哪个会话负责提取 → 存入哪个文件 → 谁审核 |
| Chat 会话读取路径文档化 | 确保 Chat 集群（claude.ai 战略参谋等）知道去哪里读数据、怎么读 |

**Phase 1 出口标准**：
- 事实圣经 v1.0 包含：角色（63+）、叙事结构（3 部）、设计决策（含被砍机制）、团队历史时间线、制作人承诺清单
- Stage 1 验证通过（日报 14 天连续有用）
- 知识捕获有标准流程，不依赖"谁记得就谁写"

### Phase 2：内容权威（6-8 月）

> Wiki 从"自动生成的骨架"变成"值得信赖的信息源"。

**2a. 数据完整性（主线 A 延续）**

| 任务 | 说明 |
|------|------|
| 角色数据补全至 100% | 63 角色全部有完整的结构化技能、数值、立绘、语音框架数据 |
| 叙事内容页 | 每章主线剧情摘要 + 角色线索追踪（利用采访披露的压缩细节作为编辑注释）|
| 联动内容预备 | 采访确认即将有"宇宙恐怖经典作品"角色联动。Wiki 需要能快速生成联动角色页 |

**2b. SEO 与分发（主线 C）**

| 任务 | 说明 |
|------|------|
| Google Search Console 接入 | 监测索引情况和搜索排名 |
| 内容去重与质量提升 | 自动生成页面增加人工审核层（至少热门角色页经过校对）|
| 社区可见性 | Reddit r/Morimens、Discord 官方频道适当提及 Wiki 存在 |

**2c. 情报循环成熟（主线 B 延续）**

| 任务 | 说明 |
|------|------|
| 接通 YouTube 数据源 | 代码就绪，配置 API Key |
| 联动事件实战测试 | 联动公布时，日报需捕获社区反应并生成有意义的摘要 |
| 周报/月报机制 | 日报之上叠加趋势分析（"本周 Steam 评论情绪变化"）|

**Phase 2 出口标准**：
- Wiki 角色页数据 100% 完整，热门角色页经过人工校对
- 联动内容在联动公布后 48 小时内上线 Wiki
- 日报系统经历过至少一次真实社区热度事件（联动/版本更新）

### Phase 3：方法论沉淀（9-10 月）

> 从"用 AI 做事"变成"总结出怎么用 AI 做事"。

| 任务 | 说明 |
|------|------|
| 方法论 v2.0 文档 | 基于 Phase 0-2 的实际经验，更新 `memory/methodology.md`。不写理论，只写"我们做了什么、结果如何、为什么" |
| 缸中之脑计划书 v2.0 | 更新 `deliverables/` 中的 PDF。加入定量数据：日报使用天数、自动化节省的时间估算、知识捕获量 |
| 案例发布 | 在适当平台（GitHub Blog、技术社区）发布"一个游戏制作人如何用 AI 建运营中台"的案例。不抽象，讲故事 |

**Phase 3 出口标准**：
- 方法论文档有定量数据支撑，不是感想
- 至少一篇外部发布的案例获得 > 100 次互动（star/like/comment）

### Phase 4：衍生创作（11-12 月 →）

> 只有在 Phase 1-3 都站稳之后才启动。

| 任务 | 说明 |
|------|------|
| 衍生游戏方向决策 | 基于事实圣经中的世界观数据 + 社区反馈数据，确定游戏类型和范围 |
| 最小可玩原型 | 利用事实圣经的角色/技能/世界观数据快速搭建原型 |
| 社区共创机制 | 利用 Wiki 的互动组件（GachaSimulator、TeamBuilder）收集玩家偏好数据 |

**Phase 4 出口标准**：可玩原型 + 10 个以上社区玩家参与测试反馈。

## 四、风险与对策

| 风险 | 概率 | 影响 | 对策 |

| 联动公布时日报系统未就绪 | 中 | 错过第一次实战测试机会 | Phase 0 在 4 月中旬前完成，给 Stage 1 留出 2-3 周验证窗口 |

| 制作人无暇参与验收 | 中 | Stage 1 验证无法推进 | 日报推送方式尽量无摩擦（邮件/Discord Bot），不需要制作人主动去仓库看 |
| 仓库体积继续膨胀 | 高 | clone 变慢，Actions 超时 | Phase 0 Discord 清理必须完成。后续建立自动监控：体积超阈值时 Issue 告警 |
| 某个 Code 会话 context 用完但任务未完成 | 高 | 进度中断，上下文丢失 | 所有关键决策写 memory/，所有进行中任务写 CONTEXT.md。新会话读完即可接续 |

## 五、成功度量

### 6 个月后（2026 年 10 月）应该看到的

1. **制作人说"日报有用"** — 不是"在跑"，是"有用"。这是唯一不可替代的主观判断
2. **事实圣经覆盖率 > 80%** — 角色、叙事、设计决策、团队历史四个维度，每个维度覆盖 80% 以上的已知信息
3. **Wiki 被外部引用** — 至少 1 个社区内容创作者（Reddit/YouTube/Discord）引用了本 Wiki 的数据
4. **方法论有一个可量化的证据** — "自动化每天为制作人节省 X 分钟"或"知识库减少了新会话启动时间 X%"
5. **仓库体积可控** — < 500MB，有自动监控

### 12 个月后（2027 年 4 月）应该看到的

1. **事实圣经成为团队内部参考资源** — 不只是 AI 用，B.I.A.V. Studio 的人类成员也在用
2. **方法论被至少一个外部团队尝试复现**
3. **衍生游戏有可玩原型**
4. **缸中之脑的存在不依赖任何单一会话或单一个人** — 包括不依赖 Light 亲自操作

## 六、与之前规划的关键差异

| 维度 | 旧理解 | 新理解 |
|------|--------|--------|
| 项目本质 | AI 运营中台 | 组织记忆系统——对抗遗忘 |
| 事实圣经 | 一个子功能 | **核心主线**——同时服务 Wiki、日报、会话启动、衍生游戏 |
| Wiki | 独立子项目 | 事实圣经的对外展示层 |
| 日报 | 管线验证目标 | 制作人的决策支持工具——省出来的时间回到游戏本身 |
| 方法论 | 要刻意写的文档 | 系统运作的自然副产品——先做对，再提取经验 |
| 衍生游戏 | 与其他子项目平行 | 依赖事实圣经成熟度，是最后启动的 |
| 成功标准 | "管线在跑" | "制作人说有用" |
| 联动事件 | 未知 | **已知的时间压力**——日报必须在联动前就绪 |
| 离线版本 | 不相关 | 缸中之脑本身就是 Studio 的"离线版本"——确保极端情况下核心能力不消失 |
| 系统架构 | 单一仓库 | **双系统**——银芯（公开层，本仓库）+ 黑池（内部层）。银芯是方法论试验场，黑池处理内部数据。数据隔离，架构共享 |
| 事实圣经边界 | 尽可能全面 | 银芯只收录公开信息。内部设计文档、未发布内容属于黑池。银芯的事实圣经 = "一个外部研究者能通过公开渠道获得的最完整的 Morimens 知识库" |
