# 核心方法论

> 以下六条是制作人 Light 在实践中提炼的 AI 协作方法论，是整个 brain-in-a-vat 项目的思想基础。

1. **发现 AI 的综合智能**：Claude 不只是写代码或回答问题的工具，它具备判断、分析、审美、策划等综合能力，可以作为制作人级别的协作伙伴。

2. **发现 Code 的自主执行能力**：Claude Code 能独立写代码、自我纠错、完成工程任务，不需要人类逐行指导。

3. **数据本身是金矿**：有了结构化的数据体系，AI 可以自动生成知识文档、wiki 页面、分析报告。数据的价值不在于存储，在于被 AI 理解和加工。

4. **知识传承体系让 AI 持续变强**：通过仓库中的 memory/ 目录建立共享知识库，每个新会话不从零开始，而是站在所有前辈会话的肩膀上。知识不会因会话结束而丢失。

5. **AI 能自我驱动**：AI 可以主动采集外部信息（社区动态、wiki 数据）并沉淀为内部知识（JSON 数据库、分析报告）。人类只需设定方向，不需要手动喂数据。

6. **多 AI 协作是可行的**：通过仓库作为共享大脑、Issue 作为任务协议、角色分工（Chat 思考 / Actions 执行 / Code 探索），多个 AI 实例可以像团队一样协作，各司其职。

# AI 多会话协作方法论

> 本文件持续沉淀协作过程中的经验和模式。

## 核心架构：双集群 + 人类调度器

```
制作人（Light）= 唯一调度器
├── Chat 集群（随开随用，无固定数量）
│   ├── 某次聊战略方向
│   ├── 某次审视仓库
│   ├── 某次研究游戏机制
│   └── ...
└── Code 集群（按子项目固定）
    ├── 主控制台 — 协调、架构决策、合并分支
    ├── Code-wiki — 游戏数据集 + Wiki 站点
    ├── Code-news — 新闻聚合 + 报告系统
    └── Code-game — 衍生游戏（规划中）
```

### Chat 集群
- **无状态**：用完即走，不保留固定会话
- **无层级**：不需要一个 Chat 管其他 Chat，人类自己就是调度器
- **通过仓库传递记忆**：产出（任务清单、分析报告等）交给 Code 主控台执行
- 可以同时开多个，各干各的

### Code 集群
- **有状态**：按子项目分工，每个子项目一个会话
- **通过分支隔离**：各自在独立分支上开发，互不干扰
- **同一子项目不要同时开两个 Code 会话**，避免冲突
- 主控制台负责协调和合并，不写业务代码

### 两个集群的连接
- 仓库是中间层（共享外脑）
- `memory/` 目录是跨会话的结构化记忆
- Chat 读仓库获取上下文，产出决策 → Code 执行决策写回仓库

## 信息持久化机制

| 机制 | 文件 | 作用 |
|------|------|------|
| 全局状态 | CLAUDE.md | 所有会话的第一份读物 |
| 项目状态 | memory/project-status.md | 各子项目进度追踪 |
| 决策记录 | memory/decisions.md | 避免决策随对话消失 |
| 局部上下文 | projects/xxx/CONTEXT.md | 子项目的工作记忆快照 |
| 资产索引 | assets/index.md | Chat 和 Code 共享的可用资源清单 |
| 背景知识 | memory/morimens-context.md | 游戏相关知识积累 |

## 信息流

```
Chat 集群 ──分析/策划──→ 制作人 ──指令──→ Code 主控台 ──协调──→ Code 子项目
                                                                    │
                                              仓库（memory/assets/projects）
                                                                    │
Chat 集群 ←──拉取数据──────────────────────────────────────────────────┘
```

## 分支管理

- **main**：稳定基线
- **claude/\<描述\>-\<ID\>**：子项目开发分支，自动生成
- 功能稳定后由主控台 PR 合并回 main
- 用户无需手动管理分支，接受默认即可

## 经验记录

### 会话替换问题
- **问题**：对话过期后，新会话需要从零了解项目
- **解决**：通过 CLAUDE.md + CONTEXT.md 体系，新会话 60 秒内进入工作状态

### 上下文污染问题
- **问题**：一个会话处理多个子项目时，上下文越来越乱
- **解决**：严格的职责隔离，一个会话一个子项目

### 知识易失问题
- **问题**：重要决策只存在于对话中，对话消失则决策丢失
- **解决**：所有决策立即写入 memory/decisions.md，git 永久保存

### 架构演进问题
- **问题**：早期决策（如"不使用框架"）随项目扩展不再适用
- **解决**：主控台定期审视架构，及时废除过时原则，记录在 decisions.md

### 分支管理问题
- **问题**：多个会话各自开分支，时间久了分支散乱、架构不同步
- **解决**：定期由主控台整理分支，统一合并到 main 建立基线

### Issue 积压问题
- **问题**：战略参谋批量创建 Issue，Actions 执行失败后 Issue 悄无声息地积压
- **根因**：没有 WIP 限制、没有失败标记、没有去重检查
- **解决**：WIP 上限 3 个/子项目 + 失败自动打 blocked 标签 + 创建前查重

## 前台专岗分类与上下文管理

每个 chat 会话按职能分类命名，模拟专业分工：

**前台（Chat）角色：**
- 战略参谋 — 架构设计、文档交付、决策讨论
- 情报官 — 社区日报、舆情分析
- 数据员 — 数据库校验、JSON 补全
- 命轮审美 — 命轮卡面设计、美术方向

**中台（Code）角色：**
- 主控台 — 全局调度、跨子项目操作
- Code-news — 聚合器开发维护
- Code-wiki — Wiki 站点 + 数据库
- Code-game — 衍生游戏（待启动）

### 核心原理

每个 chat 的上下文窗口是有限的。如果所有话题混在一个 chat 里，上下文被稀释，前面的细节被压缩或丢失。按职能分类后，每个 chat 的整个上下文窗口只包含该领域的信息，形成专业纵深。

本质上是用 chat 分类模拟专业分工，用上下文窗口模拟专业记忆——每个 chat 就是一个专岗员工的大脑，只记得自己领域的事，但记得很深。

### 两轴记忆模型

- **横向（跨 chat 共享）**：GitHub 仓库 memory/ — 所有会话都能读取的结构化记忆
- **纵向（单 chat 纵深）**：上下文窗口 — 每个专岗 chat 自然积累的领域深度记忆

横向靠仓库，纵向靠 chat 分类。两者互补，缺一不可。

### 与 Project 功能的对比

Claude 的 Project 功能是给所有对话塞同一份背景文件，上下文是扁平的。专岗分类的做法是让每个 chat 自然积累各自领域的纵深上下文，仓库只负责跨 chat 的共享记忆。专岗分类在上下文利用效率上优于 Project。

## 战略参谋实战经验

### 派 Issue 的纪律

- **先问根因**：派 Issue 前先问——这个 Issue 是在解决真实问题，还是在补救上一个 Issue 造成的问题？后者说明应该先修上游。
- **机制优于一次性**：如果一个机制性 Issue（如自动分支清理）能让一个一次性 Issue（如手动分支清理）变得不必要，优先派机制性的，不要两个都派。
- **少派多盯**：战略参谋的价值在于结果质量，不在 Issue 数量。派出去的 Issue 要持续跟踪结果，而不是派完就忘。

### 交付物受众意识

- **先确认受众再动笔**：生成任何文档/报告前先确认受众——给制作人看、给团队看、还是给外部看，语言风格完全不同。
- **默认通俗**：默认用通俗语言，技术术语只在必要时出现。受众不是工程师时，尤其要避免专业黑话。

### 能力边界意识

- **承诺前先验证可行性**：不要承诺做不到的事。例如：服务器环境无法直接调用 B站 API（需要浏览器环境），不应列为可交付承诺。
- **验证后再对外说**：对外承诺的每一项能力，都应已经过本地验证或有可靠依据。
