# AI 多会话协作方法论

> 本文件持续沉淀协作过程中的经验和模式。

## 核心架构：双集群 + 人类调度器

```
制作人（Light）= 唯一调度器
├── Chat 集群（随开随用，无固定数量）
│   ├── 某次聊战略方向
│   ├── 某次审视仓库
│   ├── 某次研究游戏机制
│   └── ...
└── Code 集群（按子项目固定）
    ├── 主控制台 — 协调、架构决策、合并分支
    ├── Code-wiki — 游戏数据集 + Wiki 站点
    ├── Code-news — 新闻聚合 + 报告系统
    └── Code-game — 衍生游戏（规划中）
```

### Chat 集群
- **无状态**：用完即走，不保留固定会话
- **无层级**：不需要一个 Chat 管其他 Chat，人类自己就是调度器
- **通过仓库传递记忆**：产出（任务清单、分析报告等）交给 Code 主控台执行
- 可以同时开多个，各干各的

### Code 集群
- **有状态**：按子项目分工，每个子项目一个会话
- **通过分支隔离**：各自在独立分支上开发，互不干扰
- **同一子项目不要同时开两个 Code 会话**，避免冲突
- 主控制台负责协调和合并，不写业务代码

### 两个集群的连接
- 仓库是中间层（共享外脑）
- `memory/` 目录是跨会话的结构化记忆
- Chat 读仓库获取上下文，产出决策 → Code 执行决策写回仓库

## 信息持久化机制

| 机制 | 文件 | 作用 |
|------|------|------|
| 全局状态 | CLAUDE.md | 所有会话的第一份读物 |
| 项目状态 | memory/project-status.md | 各子项目进度追踪 |
| 决策记录 | memory/decisions.md | 避免决策随对话消失 |
| 局部上下文 | projects/xxx/CONTEXT.md | 子项目的工作记忆快照 |
| 资产索引 | assets/index.md | Chat 和 Code 共享的可用资源清单 |
| 背景知识 | memory/morimens-context.md | 游戏相关知识积累 |

## 信息流

```
Chat 集群 ──分析/策划──→ 制作人 ──指令──→ Code 主控台 ──协调──→ Code 子项目
                                                                    │
                                              仓库（memory/assets/projects）
                                                                    │
Chat 集群 ←──拉取数据──────────────────────────────────────────────────┘
```

## 分支管理

- **main**：稳定基线
- **claude/\<描述\>-\<ID\>**：子项目开发分支，自动生成
- 功能稳定后由主控台 PR 合并回 main
- 用户无需手动管理分支，接受默认即可

## 经验记录

### 会话替换问题
- **问题**：对话过期后，新会话需要从零了解项目
- **解决**：通过 CLAUDE.md + CONTEXT.md 体系，新会话 60 秒内进入工作状态

### 上下文污染问题
- **问题**：一个会话处理多个子项目时，上下文越来越乱
- **解决**：严格的职责隔离，一个会话一个子项目

### 知识易失问题
- **问题**：重要决策只存在于对话中，对话消失则决策丢失
- **解决**：所有决策立即写入 memory/decisions.md，git 永久保存

### 架构演进问题
- **问题**：早期决策（如"不使用框架"）随项目扩展不再适用
- **解决**：主控台定期审视架构，及时废除过时原则，记录在 decisions.md

### 分支管理问题
- **问题**：多个会话各自开分支，时间久了分支散乱、架构不同步
- **解决**：定期由主控台整理分支，统一合并到 main 建立基线

## 三层工作架构（2026-03-29 更新）

```
Chat（claude.ai）──创建 Issue──→ GitHub Actions（自动 Claude Code）
                                        │
                        ┌───────────────┘
                        │ 复杂任务 / 调试 / 探索
                        ↓
                手动 Code（Claude Code CLI）
```

### 各层职责

- **Chat（claude.ai）**：制作人判断、深度分析、审美把控、创建 Issue 派活。不能 push 代码。
- **Actions（自动 Claude Code）**：执行明确任务，Issue 驱动，无需人工中转。需要 Anthropic API 余额。
- **手动 Code**：复杂交互、调试、探索。有持续上下文，能记住会话内过程。

### 选择哪层

| 场景 | 推荐层 |
|------|--------|
| 策略分析、审美判断、Issue 撰写 | Chat |
| 明确的文件增删改任务 | Actions |
| 调试、多轮交互、架构探索 | 手动 Code |

## 上下文传承

- 手动 Code 会话销毁后上下文丢失，但仓库 `memory/` 目录保存了所有重要知识
- 新 Code 会话启动时读 `CLAUDE.md` → `memory/` 即可在五分钟内接上
- 仓库是不会被稀释的外挂记忆，比 Chat 的内置记忆系统可靠
- Actions 会话每次都是全新启动，完全依赖仓库 memory 获取上下文

## GitHub Token 与密钥运维

- **Fine-grained PAT**（最长1年）：限单仓库 + Issues 读写，用于 Chat 创建 Issue
- **ANTHROPIC_API_KEY**：存为仓库 Secret，供 Actions 调用 Claude API（需账户有余额）
- **Claude GitHub App**：需安装并授权，Actions 才能正常运行
- Token 出现在对话文本中有泄露风险，建议用完 revoke 重新生成
- Actions 失败时先查：余额 → Secret 配置 → App 权限，按顺序排查
