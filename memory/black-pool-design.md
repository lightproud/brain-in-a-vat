# 缸中之脑·黑池（Black Pool）架构设计方案

> 最后更新：2026-04-02 by 战略中心（Code）
> 设计者：战略中心（Code）
> 日期：2026-04-01
> 状态：方案已确认（2026-04-02 制作人确认黑池为内网版本）
> 信息边界：本文件本身是公开的（存在银芯仓库），但黑池的实际内容不公开
> 形态：公司内网系统（非 GitHub 私有仓库）

---

## 一、定位

黑池是缸中之脑的内部信息层，处理不适合公开的数据：

| 内容类型 | 示例 |
|----------|------|
| 内部设计文档 | 未发布角色设定、技能数值表、关卡设计稿 |
| 商业数据 | 收入数据、付费率、留存率、LTV |
| 未发布内容 | 第二部/第三部剧情大纲、未公开联动对象 |
| 内部决策记录 | 团队人事、合同、商业谈判 |
| 玩家数据分析 | 行为数据、流失分析、付费画像 |
| 内部沟通记录 | 团队会议纪要、跨部门协调 |

**银芯绝对不能出现的内容**：未发布内容剧透、商业敏感数据、玩家个人信息、内部人事信息。

---

## 二、架构设计原则

### 2.1 银芯验证，黑池复用

银芯是方法论试验场。所有在银芯验证过的模式，黑池直接复用：

| 银芯验证的模式 | 黑池复用方式 |
|---------------|-------------|
| memory/ 结构化记忆 | 同结构，内容换为内部决策、商业复盘 |
| CLAUDE.md 会话引导 | 同模式，角色定义换为内部职能 |
| assets/data/ 事实圣经 | 同结构，数据换为内部设计文档和数值表 |
| 日报自动生成 | 同管线，数据源换为内部数据仪表盘 |
| 知识捕获流程 | 同流程，捕获对象换为内部会议和设计评审 |

### 2.2 数据完全隔离

- 银芯 = GitHub 公开仓库；黑池 = 公司内网系统（非 GitHub 仓库）
- 两者不共享 git 历史，数据物理隔离
- 黑池不引用银芯的文件路径（避免依赖），但可引用银芯的公开 URL
- 银芯不知道黑池的存在细节（本文件除外）

### 2.3 单向信息流

```
黑池（内部）──筛选/脱敏──→ 银芯（公开）
                             ↑
                       绝不反向流动
```

例如：内部设计了一个新角色 → 角色公开发布后 → 银芯的 Wiki 和事实圣经才录入公开信息。

## 三、目录结构

复用银芯的目录范式，但内容完全不同：

```
black-pool/
├── CLAUDE.md                    # 黑池会话引导（角色定义偏内部职能）
├── memory/
│   ├── project-status.md        # 游戏开发内部进度（含未发布内容排期）
│   ├── decisions.md             # 内部决策日志（含商业决策）
│   ├── team-status.md           # 团队状态（人员、产能、瓶颈）
│   ├── postmortems.md           # 复盘记录（版本上线后复盘）
│   └── risk-register.md         # 风险登记簿（商业/技术/人员风险）
├── assets/
│   ├── data/
│   │   ├── unreleased/          # 未发布内容设计文档
│   │   ├── analytics/           # 商业数据分析
│   │   ├── balance/             # 数值平衡表
│   │   └── roadmap/             # 内部版本路线图（黑池内网目录，非银芯路径）
│   └── templates/               # 内部文档模板
├── projects/
│   ├── game-dev/                # 游戏开发管理（排期、里程碑、阻塞项）
│   ├── monetization/            # 商业化设计与分析
│   ├── community-ops/           # 社区运营内部策略
│   └── hr/                      # 团队建设（招聘、培训、考核）
└── reports/
    ├── daily/                   # 内部日报（含商业数据）
    ├── weekly/                  # 周报
    └── monthly/                 # 月报 + 版本复盘
```

## 四、会话角色定义

黑池的会话集群与银芯独立，但结构类似：

| 角色 | 职责 | 信息权限 |
|------|------|----------|
| 黑池·主控 | 内部项目管理、跨部门协调、排期追踪 | 全部 |
| 黑池·数值 | 战斗数值平衡、养成曲线、经济系统 | 设计文档 + 数值表 |
| 黑池·运营 | 活动策划、商业化分析、玩家行为分析 | 商业数据 + 社区数据 |
| 黑池·叙事 | 剧情大纲管理、角色设定归档、世界观一致性检查 | 未发布叙事内容 |

**关键区别**：银芯的会话可以公开讨论。黑池的会话内容不应出现在任何公开渠道。

## 五、核心功能

### 5.1 内部日报

银芯日报监测社区外部情绪。黑池日报监测内部状态：

```markdown
# B.I.A.V. Studio 内部日报 2026-04-02

## 开发进度
- 第二部第五章：脚本 80% → 美术 40% → 战斗设计 20%
- 下一个限定角色 [内部代号]：技能设计完成，数值调试中
- 联动角色 [内部代号]：美术定稿，等待授权方确认

## 数值警报
- ⚠️ [角色名] 上线后 3 天胜率数据异常（85%），可能需要环境调整
- 本周新增付费用户 XXX，ARPU ¥XX.X

## 社区信号（从银芯日报同步）
- Steam 好评率 83%，无异常
- Discord 活跃度正常（11.8K 消息/天）
- 无跨平台负面趋势

## 本周阻塞项
- 美术产能：新角色立绘延期 2 天（原画师请假）
- 联动授权：等待对方法务回复（已催促）

## 需要制作人决策
- [角色名] 数值调整方案 A/B/C，详见 assets/data/balance/xxx.json
```

### 5.2 版本复盘自动化

每个版本上线后 7 天，自动汇总：
- 银芯社区反馈数据（自动从银芯日报聚合）
- 内部商业数据（付费率、留存变化）
- 数值表现（角色使用率、胜率、通关率）
- 生成结构化复盘文档存入 `reports/monthly/`

### 5.3 知识传承

采访中提到团队人员频繁流动。黑池的核心价值之一是：

- 新成员入职时，读 `CLAUDE.md` + `memory/` 即可了解当前项目状态、历史决策和设计原则
- 成员离职前，关键知识通过 AI 会话提取并写入 `assets/data/`
- 设计评审记录结构化存入 `memory/decisions.md`

这就是"拉蒙娜机制"在团队管理层面的实现。

### 5.4 银芯信号接入

黑池应该能读取银芯日报的关键信号（不反向）：

```
银芯日报 → 导出 JSON → 黑池读取 → 内部日报"社区信号"区块
```

实现方式：
- 方案 A：黑池 clone 银芯的 `projects/news/output/` 目录（只读）
- 方案 B：银芯日报同时写一份 JSON 摘要到固定 URL，黑池 fetch
- 方案 C：银芯仓库设为 public（已是），黑池直接 raw.githubusercontent.com 读取

推荐**方案 C**——最简单，银芯本身就是公开的。

## 六、启动条件

黑池不应该现在就启动。启动条件：

| 条件 | 当前状态 | 说明 |
|------|----------|------|
| 银芯 Stage 1 验证通过 | 未通过（进行中） | 方法论需要先在银芯验证 |
| 制作人确认有内部数据需要管理 | 待确认 | 是否有足够的内部数据值得建系统 |
| ANTHROPIC_API_KEY 有余额 | 无 | 黑池的 AI 会话需要 API |
| 制作人确认隐私边界 | 待确认 | 哪些数据可以进黑池，哪些太敏感连 AI 也不能看 |

**建议启动时间**：银芯 Phase 1（记忆宫殿）完成后，即 2026 年 6 月左右。

## 七、隐私与安全

### 必须遵守

1. 黑池仓库必须为 **private**
2. 不使用第三方 CI/CD 处理黑池数据（避免数据泄露到第三方日志）
3. 黑池的 CLAUDE.md 必须明确标注"本仓库内容为公司内部机密"
4. AI 会话的上下文不应包含玩家个人身份信息（PII）
5. 商业数据（收入、LTV 等）只以趋势和比率呈现，不出现绝对数字（降低泄露影响）

### 需要讨论

1. **AI 能看哪些数据？** — 未发布角色设定可以？收入数据可以？合同细节？制作人需要划一条明确的线
2. **黑池是否需要多人访问？** — 如果只有制作人一个人用，安全要求低很多。如果团队成员也用，需要权限分层
3. **Claude 平台记忆 vs 仓库存储** — 某些极敏感信息（如密码、合同金额）是否只存在 Claude 平台记忆中，不写入仓库文件？

## 八、与银芯的协作流程

```
内部设计完成 → 黑池归档设计文档
         ↓
内容公开发布 → 银芯录入公开信息到事实圣经
         ↓
社区反馈产生 → 银芯日报采集
         ↓
银芯日报信号 → 黑池内部日报引用
         ↓
内部复盘决策 → 黑池记录 → 影响下一轮设计
```

这就是完整的信息循环。银芯面向外部（社区、公开信息），黑池面向内部（开发、商业），两者通过日报信号单向连接。

## 九、最小可行版本（MVP）

如果要最快启动黑池，只需要：

1. 创建 private repo `brain-in-a-vat-internal`（或其他名字）
2. 复制银芯的 `CLAUDE.md` 模板，改写角色定义和信息边界
3. 创建 `memory/` 目录，写入当前已有的内部决策（从制作人脑子里提取）
4. 建立内部版本路线图管理

总工作量：一个 Code 会话 + 制作人 1 小时口述内部状态 → AI 结构化录入。

其余功能（内部日报、数值监控、复盘自动化）在 MVP 稳定后逐步加。
