# 缸中之脑内部 AI 工具平台 — 下一阶段建设方案

## Context

制作人 Light 提出将 BIAV 从个人 AI 终端扩展为团队级内部 AI 工具平台，替代 Qoder，核心逻辑：
- 企业 API Key 统一管理（安全合规 + 成本可控）
- 黑池（Black Pool）作为团队集体知识层，是核心护城河
- 先从**配置表和外围提效工具**切入，不碰专业程序员的代码编辑场景
- 支持非灵犀团队成员（>50% 开发团队当前无法使用 AI）
- 统一管线：信息采集 → Bug 提取 → 自动定位 → 修复建议
- 需要找/培养一名 AI 工具工程师协同推进

当前资产：BIAV Web Terminal v0.14.0（3,679 行单文件）、银芯 9 模块记忆系统、13 条 GitHub Actions、黑池已初始化（21 文件）。

---

## 推荐方案：三阶段渐进式建设

### Phase A：基座层（2-3 周）
> 目标：让团队能安全地用上 AI，制作人能看到用量

#### A1. 企业 API 网关
- **做什么**：部署一个轻量 API 代理（Python FastAPI / Node），统一转发 Anthropic/OpenAI 请求
- **核心功能**：
  - Token 认证（每人一个 token，不暴露原始 API Key）
  - 用量记录（who + when + model + tokens_in/out）
  - 成本限额（按人/按组/按日设上限，超限返 429）
  - 请求日志（可审计，敏感内容可选脱敏）
- **部署**：内网服务器 or Docker，黑池 SVN 管理代码
- **文件**：新建 `black-pool/api-gateway/` 项目

#### A2. 多用户 BIAV
- **做什么**：BIAV Web Terminal 增加登录 + 权限分层
- **改动点**（`projects/biav/index.html`）：
  - 登录页：用户名 + 网关 Token（替代直接输入 API Key）
  - API 请求改走网关地址而非直连 Anthropic
  - 管理员面板：用量仪表盘、用户管理、全局设置
  - 对话数据隔离（当前 localStorage 按用户前缀）
- **不改**：核心对话逻辑、工具系统、记忆系统保持不变

#### A3. 黑池知识注入
- **做什么**：网关层在系统提示词中自动注入黑池知识
- **机制**：
  - 网关拦截请求 → 根据用户角色注入对应知识片段
  - 知识片段来源：黑池 `knowledge/` 目录（手动维护）
  - 角色映射：QA → 测试规范+已知 Bug 库；策划 → 设计文档+决策记录；运营 → 社区数据+话术库
- **效果**：团队成员无需手动提供上下文，AI 自带项目知识

---

### Phase B：工种工具层（4-6 周）
> 目标：为每个工种提供专用 AI 工具，从配置表和外围场景切入

#### B1. 配置表助手（首个落地工具）
- **场景**：策划修改数值配置（角色属性、技能参数、关卡数据）
- **功能**：
  - 上传 Excel/JSON 配置表 → AI 校验格式和逻辑（如数值范围、引用完整性）
  - 自然语言修改指令 → 生成 diff 预览 → 确认后导出
  - 批量生成：「生成 10 个难度递增的关卡配置」
  - 与黑池数据校验联动（引用角色 ID 是否存在等）
- **实现**：BIAV 新增专用项目模式（类似现有的 6 个预设模式）
- **文件**：`projects/biav/index.html` 新增工具 + `black-pool/tools/config-validator/`

#### B2. QA 测试助手
- **场景**：QA 提交 Bug 报告、回归测试
- **功能**：
  - 粘贴日志/截图 → AI 自动分类（UI/逻辑/性能/崩溃）+ 建议优先级
  - 关联已知 Bug 库（黑池维护）→ 判断是否重复
  - 生成标准化 Bug 报告模板
  - 回归测试 checklist 自动生成
- **实现**：BIAV 项目模式 + 黑池 Bug 知识库

#### B3. 翻译/本地化助手
- **场景**：多语言文本翻译和校对
- **功能**：
  - 批量翻译（保留格式标记和变量占位符）
  - 术语一致性检查（黑池维护术语表）
  - 语境感知：注入角色设定和世界观，确保翻译风格一致
- **实现**：BIAV 项目模式 + 黑池术语库

#### B4. 运营内容助手
- **场景**：社区管理、公告撰写、舆情分析
- **功能**：
  - 银芯社区数据自动注入 → 每日舆情摘要
  - 公告/回复模板生成（多平台适配：Steam/B站/Discord）
  - 敏感内容检测预警
- **实现**：复用银芯现有新闻管线 + BIAV 运营模式

---

### Phase C：自动化管线层（6-8 周）
> 目标：从被动工具升级为主动管线，实现 RSShub 式统一信息流

#### C1. 统一信息采集管线
- **架构**：扩展银芯现有 3 源采集 → 内部源（SVN commit log、内部 IM、测试平台）
- **流程**：采集 → 结构化 → 黑池存储 → 按角色推送
- **复用**：`scripts/dream.py` 哨兵层异常检测模式

#### C2. Bug 自动定位管线
- **流程**：QA 报告 → 日志分析 → 代码搜索 → 生成定位报告
- **注意**：只输出分析报告和修复建议，**不自动修改代码**
- **依赖**：黑池需接入内部代码仓库（只读）

#### C3. 做梦 Agent 团队版
- **复用**：银芯 `scripts/dream.py` 三层架构（浅睡/深睡/REM）
- **扩展**：
  - 浅睡：每日自动汇总各工种 AI 使用情况 + 成本
  - 深睡：跨工种知识关联（QA 发现的 Bug 模式 → 策划配置建议）
  - REM：周度洞察报告给制作人

## 人员计划

| 角色 | 职责 | 来源 |
|------|------|------|
| Light（制作人） | 架构决策、工种需求定义、品质标准 | 现有 |
| AI 工具工程师（1人） | 网关开发、BIAV 功能扩展、管线维护 | 招聘/培养 |
| 各工种代表（兼职） | 需求反馈、工具验收、知识维护 | 现有团队 |

**AI 工具工程师画像**：
- 全栈能力（Python + 前端），不需要顶尖但要能独立交付
- 理解 AI API 调用链（prompt engineering、tool use、streaming）
- 关键：愿意深入理解业务场景，不是纯技术导向
- Light 负责架构和品质把关，工程师负责实现和维护

## 技术选型建议

| 组件 | 推荐 | 理由 |
|------|------|------|
| API 网关 | Python FastAPI | 与现有银芯脚本栈一致，部署简单 |
| 用量存储 | SQLite → 后期可迁 PostgreSQL | 初期够用，无运维负担 |
| 前端 | 继续单文件 BIAV | 已验证架构，0.14.0 功能完备 |
| 知识存储 | 黑池 SVN + JSON/Markdown | 与现有模式一致 |
| 部署 | 内网 Docker Compose | 网关 + 静态文件服务一体化 |

## 里程碑与验收标准

| 里程碑 | 验收标准 | 预计节点 |
|--------|----------|----------|
| **M1: 团队能用** | 5+ 人通过网关使用 BIAV，管理员能看到用量 | Phase A 完成 |
| **M2: 配置表落地** | 策划用 AI 完成一次真实配置表修改，效率提升可感知 | Phase B1 完成 |
| **M3: 三工种覆盖** | QA/翻译/运营各有专用模式且在日常使用 | Phase B 完成 |
| **M4: 主动推送** | 团队每天收到自动生成的相关摘要 | Phase C1 完成 |

---

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| 团队不愿用 / 学习成本 | Phase A 先让 3-5 个愿意尝试的人用起来，用口碑带动 |
| 维护成本超预期 | 网关和 BIAV 都走最简路径，不过度工程化 |
| AI 工具工程师找不到 | 先由 Light + Claude Code 推进 Phase A，Phase B 时必须到位 |
| 黑池知识维护负担 | 初期只维护高频知识（配置表规范、术语表），不追求全量 |
| 内网部署权限受限 | 提前确认 IT 部门支持，备选方案：云服务器 + VPN |

## 验证方式

- Phase A 验证：部署网关后，用 curl 测试 token 认证 + 用量记录 + 限额拦截
- BIAV 多用户验证：两个浏览器分别登录不同账号，确认对话数据隔离
- Phase B 验证：拿一张真实配置表走完「上传 → AI 校验 → 修改 → 导出」全流程
- 成本验证：运行一周后对比网关日志，确认 token 消耗在预算内
