1. 背景与动机

当前 AI 辅助开发的问题

大多数人使用 AI 写代码的方式是单对话模式：一个对话窗口处理所有事情。随着项目复杂度增长，这种方式会遇到三个瓶颈：

问题 表现 后果
上下文溢出 对话越来越长，AI 遗忘早期内容 重复解释需求，决策前后矛盾
职责混乱 前端、后端、部署全在一个对话里 频繁切换上下文，效率低下
知识易失 对话过期/存档后，上下文全部丢失 新对话需要从零开始，无法接手

核心矛盾：项目需要持久、有组织的知识管理，但 AI 对话天然是临时性的、线性的。

2. 核心思想

解决方案借鉴了软件工程中的微服务架构思想：

把一个项目拆成多个子项目，每个子项目由一个独立的 AI 会话负责，再用一个"主控制台"会话统一协调。

就像管理一个真实的开发团队：

传统团队 AI 多会话模式
项目经理 主控制台会话
前端开发 前端子项目会话
后端开发 后端子项目会话
项目文档 CLAUDE.md + CONTEXT.md 文件
站会/周会 主控制台读取并更新状态
新人入职文档 新会话指引（写在 CLAUDE.md 中）

三个核心原则

1职责隔离 — 每个会话只负责一个明确的子项目，不越界

2状态持久化 — 关键信息写入仓库文件，而非留在对话中

3可替换性 — 任何会话过期后，新会话能通过文件快速接手

3. 架构设计

整体架构图

目录结构

project/
├── CLAUDE.md              # 全局协调文件（所有会话共读）
├── README.md              # 项目对外说明
├── sub-project-a/         # 子项目 A
│   ├── CONTEXT.md         # A 的局部上下文
│   └── ...                # A 的代码
├── sub-project-b/         # 子项目 B
│   ├── CONTEXT.md         # B 的局部上下文
│   └── ...                # B 的代码
└── sub-project-c/         # 子项目 C
    ├── CONTEXT.md         # C 的局部上下文
    └── ...                # C 的代码

分支策略

每个子项目在独立的 Git 分支上开发，由主控制台决定合并：

claude/feature-a-xxxxx   ← 子项目 A 的工作分支
claude/feature-b-xxxxx   ← 子项目 B 的工作分支
claude/main-control-xxx  ← 主控制台的工作分支

4. 关键机制

4.1 CLAUDE.md — 全局协调中枢

这是整个架构的核心文件。Claude Code 在启动时会自动读取仓库根目录的 CLAUDE.md，因此它天然适合作为跨会话的信息中枢。

CLAUDE.md 应包含：

板块 内容 作用
项目愿景 一句话说清楚项目要做什么 对齐所有会话的方向
目录结构 各子项目的目录和文件说明 新会话快速定位
子项目状态 每个子项目的当前进度和待办 状态同步
协作规范 分支命名、代码风格、提交规范 统一标准
新会话指引 新会话启动后该做什么 零成本接手
决策记录 重要的架构/技术决策及原因 知识持久化
变更日志 按时间记录重大变更 追踪演进

4.2 CONTEXT.md — 子项目局部上下文

每个子项目目录下放置一个 CONTEXT.md，记录该子项目特有的信息：

  • 这个子项目负责什么
  • 文件和模块说明
  • 当前状态和待办事项
  • 技术细节和注意事项

信息分层原则：全局信息放 CLAUDE.md，局部信息放 CONTEXT.md。新会话先读全局，再读自己负责的局部。

4.3 主控制台的职责

主控制台不写业务代码，它的职责是：

  1. 项目规划 — 决定做什么、怎么拆分子项目
  2. 架构决策 — 技术选型、目录结构、规范制定
  3. 状态追踪 — 更新 CLAUDE.md 中的子项目状态
  4. 代码审查 — 检查子项目的产出质量
  5. 问题诊断 — 当子项目遇到跨模块问题时介入

5. 实践案例

项目：忘却前夜 Morimens 情报平台

一个游戏社区情报收集与衍生内容开发项目，采用本方法论进行 AI 协作开发。

会话分工

会话 角色 职责 目录
主控制台 总控制 项目规划、架构设计、协调各子项目 根目录
新闻收集器 子项目 多平台社区热点自动聚合 news/
官方数据库 子项目 游戏数据系统性整理 database/
衍生游戏 子项目 同人游戏开发（规划中） game/

实际成果

  • 新闻收集器：完整的前端页面 + Python 抓取脚本 + GitHub Actions 自动化，全程 AI 编写，零手写代码
  • 主控制台：完成项目架构重组、目录分离、协调文件体系建设
  • 技术栈：纯 HTML/CSS/JS + Python + GitHub Pages，零运维成本

协作流程示例

1. 主控制台规划项目 → 写入 CLAUDE.md 2. 创建"新闻收集器"会话 → 读取 CLAUDE.md → 在 news/ 下开发 3. 新闻收集器完成功能 → 更新 CONTEXT.md → 提交到独立分支 4. 主控制台审查代码 → 确认合并 → 更新 CLAUDE.md 状态 5. 创建"官方数据库"会话 → 读取 CLAUDE.md → 在 database/ 下开发 6. 循环...

6. 操作指南

从零开始搭建

1创建仓库和主控制台

创建一个 Git 仓库，打开第一个 Claude Code 对话作为主控制台。让它创建 CLAUDE.md，定义项目愿景和整体架构。

2规划子项目

在主控制台中明确要拆分哪些子项目，创建对应目录，在每个目录下放置 CONTEXT.md。

3启动子项目会话

为每个子项目新建一个 Claude Code 对话，关联到同一个仓库。子项目会话启动后会自动读取 CLAUDE.md，了解全局再开始工作。

4开发与同步

各子项目在自己的分支和目录下独立开发。完成阶段性成果后更新 CONTEXT.md。主控制台定期检查并更新 CLAUDE.md 中的全局状态。

5会话替换

当某个子项目会话过期或需要重启时，新建对话即可。新会话通过 CLAUDE.md + CONTEXT.md 快速了解项目状态，无缝接手。

最佳实践

  • 决策不留在对话里 — 重要决策立即写入 CLAUDE.md 的决策记录
  • 一个会话一个职责 — 不要让子项目会话处理跨模块的事
  • 主控制台不写业务代码 — 只做规划、协调、审查
  • 及时更新状态 — 每完成一个阶段就更新对应的 .md 文件
  • 分支隔离 — 各子项目在独立分支开发，避免冲突

7. 已知局限与优化方向

当前局限

局限 说明 可能的优化
无实时通信 会话之间只能通过文件异步同步 未来可能通过 MCP 或 API 实现会话间通信
人工调度 主控制台无法自动触发子项目会话 结合 CI/CD 实现自动化触发
会话不持久 对话过期后需要手动重建 通过完善的文件体系降低重建成本
状态同步延迟 需要手动 git pull 才能看到其他会话的更新 约定更频繁的提交和推送节奏

演进方向

  1. 自动化调度 — 通过 GitHub Actions 或脚本自动触发 AI 会话执行任务
  2. 状态看板 — 自动从各 CONTEXT.md 生成项目状态仪表板
  3. 模板化 — 将这套方法论抽象为可复用的项目模板
  4. 质量门禁 — 子项目提交自动触发主控制台审查

8. 总结

一句话概括：用文件系统作为 AI 会话之间的"共享记忆"，用目录结构实现职责隔离，用主控制台会话代替项目经理，让一个人可以驱动一个 AI 开发团队。

适用场景

  • 个人开发者想用 AI 完成中大型项目
  • 项目涉及多个独立模块（前端 + 后端 + 数据 + 部署等）
  • 项目周期较长，需要持久化知识和决策
  • 需要频繁切换或重启 AI 对话

不适用场景

  • 简单的单文件脚本或小工具
  • 一次性对话就能完成的任务
  • 不使用 Git 的项目

关键收益

收益 说明
效率提升 每个会话专注一件事，上下文精准，减少重复解释
知识不丢失 决策和状态持久化到文件，不随对话消失
可扩展 需要新模块时，创建新目录 + 新会话即可
低门槛 不需要额外工具，Git + Claude Code 即可运行
